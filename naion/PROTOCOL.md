# Naion CSM Protocol

This document describes the unified runtime packet protocol implemented by:

- `csm.h`
- `csm.py`
- `csm.go`

It is the client/server secure messaging protocol used for runtime communication, distinct from the config-file encryption flow.

## Overview

CSM means client-server-model.

Trust model:

- The client holds the server Ed25519 public key.
- The server does not know the client public key before the first client packet is decrypted.
- The first valid client packet reveals the client Ed25519 public key inside the encrypted plaintext.
- After that, the server can encrypt response packets back to that client identity.

Cryptographic building blocks:

- Identity/signature: Ed25519 detached signature
- Key agreement: X25519
- Payload encryption: libsodium-compatible `crypto_box_easy` / `naion_box_easy`

## Packet Layout

The outer packet layout is the same in both directions:

```text
signature(64) | session_x25519_public_key(32) | nonce(24) | mac_and_ciphertext(variable)
```

Field meanings:

- `signature`: Ed25519 detached signature over everything after it
- `session_x25519_public_key`: one-time ephemeral X25519 public key for this packet
- `nonce`: 24-byte box nonce
- `mac_and_ciphertext`: `crypto_box_easy` / `naion_box_easy` output in combined `mac || ciphertext` order

Combined-layout note:

- CSM intentionally uses the standard `crypto_box_easy`/`naion_box_easy` combined layout: `mac || ciphertext`.

Fixed outer overhead:

```text
64 + 32 + 24 + 16 = 136 bytes
```

## Plaintext Layout Before Encryption

### Client -> Server

```text
packet_meta(16) | client_ed25519_public_key(32) | application_payload(variable)
```

### Server -> Client

```text
packet_meta(16) | application_payload(variable)
```

This direction asymmetry is intentional:

- The client already knows the server public key from config.
- The server learns the client public key from the first decrypted client packet.
- The server response packet therefore does not need to include the client public key again.

## PacketMeta

`PacketMeta` is exactly 16 bytes.

Binary layout, little-endian:

```text
magic[4]         = "IFW1"
protocol_ver[1]  = 1
reserved[1]      = 0
flags[2]         = uint16 little-endian
timestamp_ms[8]  = uint64 little-endian
```

Semantics:

- `magic`: identifies this packet format
- `protocol_ver`: protocol version for runtime secure messaging
- `reserved`: reserved for future use, currently 0
- `flags`: bit flags reserved for future use, currently 0
- `timestamp_ms`: Unix time in milliseconds, generated by sender

Current protocol version:

```text
1
```

## Encryption Flow

### Client Encrypt

1. Build `PacketMeta` with current `timestamp_ms`.
2. Build plaintext as:
   `packet_meta || client_ed25519_public_key || payload`
3. Convert server Ed25519 public key to X25519 public key.
4. Generate one ephemeral X25519 keypair for this packet.
5. Derive shared key using:
   `X25519(session_x25519_secret_key, server_x25519_public_key)`
6. Encrypt plaintext with `crypto_box_easy` / `naion_box_easy` using a random 24-byte nonce.
7. Build body as:
   `session_x25519_public_key || nonce || mac_and_ciphertext`
8. Sign the body with the client Ed25519 secret key.
9. Output final packet as:
   `signature || body`

### Server Decrypt

1. Split packet into `signature` and `body`.
2. Read `session_x25519_public_key` from `body`.
3. Convert server Ed25519 secret key to X25519 secret key.
4. Derive shared key using:
   `X25519(server_x25519_secret_key, session_x25519_public_key)`
5. Decrypt `nonce || mac_and_ciphertext`.
6. Parse `PacketMeta`.
7. Read `client_ed25519_public_key` from decrypted plaintext.
8. Verify detached signature over `body` using the client Ed25519 public key.
9. Store this client public key for future responses.
10. Return the remaining application payload.

### Server Encrypt

1. Ensure a client public key has already been learned from a previous client packet.
2. Build `PacketMeta` with current `timestamp_ms`.
3. Build plaintext as:
   `packet_meta || payload`
4. Convert stored client Ed25519 public key to X25519 public key.
5. Generate one ephemeral X25519 keypair for this packet.
6. Derive shared key using:
   `X25519(session_x25519_secret_key, client_x25519_public_key)`
7. Encrypt plaintext with `crypto_box_easy` / `naion_box_easy`.
8. Build body as:
   `session_x25519_public_key || nonce || mac_and_ciphertext`
9. Sign the body with the server Ed25519 secret key.
10. Output final packet as:
   `signature || body`

### Client Decrypt

1. Split packet into `signature` and `body`.
2. Verify detached signature over `body` using the server Ed25519 public key.
3. Read `session_x25519_public_key` from `body`.
4. Convert client Ed25519 secret key to X25519 secret key.
5. Derive shared key using:
   `X25519(client_x25519_secret_key, session_x25519_public_key)`
6. Decrypt `nonce || mac_and_ciphertext`.
7. Parse `PacketMeta`.
8. Return the remaining application payload.

## Directional Identity Rules

Client side:

- Must know the server Ed25519 public key before sending packets.
- Signs outbound packets with the client Ed25519 secret key.
- Verifies inbound packets with the server Ed25519 public key.

Server side:

- Does not need client public key before the first inbound packet.
- Learns client Ed25519 public key from the first decrypted client packet.
- Verifies inbound client packets with the learned client public key from that packet.
- Uses the stored client public key to encrypt later responses.

## Size Budget

UDP packet budget is capped at 1024 bytes.

Outer fixed overhead:

```text
136 bytes
```

`PacketMeta` size:

```text
16 bytes
```

Client->Server extra public key field:

```text
32 bytes
```

Maximum application payload sizes:

- Client -> Server: `1024 - 136 - 16 - 32 = 840 bytes`
- Server -> Client: `1024 - 136 - 16 = 872 bytes`

Reference constants used by the implementations:

- `MAX_UDP_DATAGRAM_BYTES = 1024`
- `MAX_CLIENT_PAYLOAD_BYTES = 840`
- `MAX_SERVER_PAYLOAD_BYTES = 872`

Oversized plaintext must not be sent through this UDP packet format.

## Validation Rules

A receiver should reject a packet when any of the following is true:

- total packet size is below the minimum expected size
- signature verification fails
- AEAD decryption fails
- `PacketMeta.magic` is not `IFW1`
- `PacketMeta.protocol_ver` is not `1`
- the client packet plaintext is too short to contain `client_ed25519_public_key`
- the server attempts to encrypt before learning a client public key
- plaintext exceeds the direction-specific UDP payload budget

## Cross-Language Notes

`csm.h`, `csm.py`, and `csm.go` are intended to implement the same high-level protocol.

Language-specific differences may exist in internal helper APIs, but these protocol rules must stay consistent:

- same outer packet layout
- same `PacketMeta` format
- same client/server plaintext direction rules
- same detached-signature coverage
- same UDP size budget

## Server Validation Policy

The packet format itself only carries `timestamp_ms`.
Replay rejection and time-freshness checks are implemented as server-side policy on top of the packet format.

### Validation Order

Recommended server receive pipeline:

1. check minimum packet size
2. decrypt packet
3. parse `PacketMeta`
4. extract client public key from client packet plaintext
5. verify detached signature
6. validate timestamp window
7. check replay cache
8. accept and process payload

The current Go and Python server helpers follow this order.

### Timestamp Window

Purpose:

- reject packets that are too old
- reject packets that are too far in the future
- reduce replay usefulness when a packet is captured and replayed later

Recommended rule:

```text
abs(server_now_ms - packet.timestamp_ms) <= allowed_window_ms
```

If this check fails, the packet should be rejected before application processing.

Current helper support:

- Go: `Server.SetTimestampWindow(time.Duration)`
- Python: `Server.set_timestamp_window_ms(int)`

Behavior:

- zero or unset window means timestamp validation is disabled
- non-zero window enables strict freshness validation

Recommended starting values:

- LAN / stable clocks: `30s` to `60s`
- general enterprise deployment: `120s`
- high-latency / poor clock environments: `180s` to `300s`

If clients may have bad local clocks, prefer a slightly larger window first, then tighten it after rollout data is available.

### Replay Cache Interface

Purpose:

- reject the same authenticated packet when replayed within the cache retention window

The replay cache is a server-side hook.
The protocol does not require a specific storage implementation.

Current helper support:

- Go:
  - interface: `ReplayCache`
  - default implementation: `MemoryReplayCache`
  - hook: `Server.SetReplayCache(...)`
- Python:
  - protocol: `ReplayCache`
  - default implementation: `MemoryReplayCache`
  - hook: `Server.set_replay_cache(...)`

Current default cache keying strategy in Go and Python:

```text
sha256(client_public_key || signature)
```

This means replay detection is tied to the authenticated client identity and exact signed packet body.

### Replay Retention

Replay cache entries should expire automatically.

Recommended retention:

- at least as long as the timestamp acceptance window
- often `2x` to `3x` the timestamp window is a good operational default

Current default in the helper memory caches:

```text
5 minutes
```

### Recommended Configurations

Recommended baseline for production rollout:

- timestamp window: `120s`
- replay cache: enabled
- replay retention: `300s`

Recommended stricter profile for well-managed endpoints:

- timestamp window: `30s` to `60s`
- replay cache: enabled
- replay retention: `120s` to `180s`

Recommended compatibility-first profile during migration:

- timestamp window: `180s` to `300s`
- replay cache: enabled
- log rejected packets
- tighten window later based on observed drift

### Operational Notes

- A replay cache should run after signature verification, not before.
- Timestamp validation alone does not stop rapid in-window replay.
- Replay cache alone does not reject very old packets if entries have expired.
- Using both together is the recommended deployment mode.
- For multi-process or multi-node servers, an in-memory cache only protects each process independently. A shared cache is needed for cluster-wide replay rejection.

## Not Covered Here

This document does not define:

- multi-client server session table behavior
- transport binding to HTTP, HTTPS, or raw UDP socket handling

Those are service-side policy decisions layered on top of this packet format.
